## Motivation

大型语言模型（LLM）通常通过prompt进行控制，但这种方式存在效率问题，每次新查询都需要重复处理整个prompt。
此外，prompt的模块化复用也面临挑战，难以将特定任务逻辑封装并独立使用。
这限制了LLM在需要大量可重用、可组合任务场景中的应用。

## Contribution

- **程序记忆框架TokMem**: 引入一个将可重用任务程序编译为单个可训练记忆token的框架。
- **恒定开销的行为控制**: 每个记忆token既作为程序索引，又作为生成控制信号，以恒定大小的开销引导生成，实现特定行为。
- **冻结LLM骨干**: TokMem在添加新程序时保持LLM骨干模型冻结，将程序知识完全存储在这些专用单元中，避免干扰现有知识。
- **优于RAG和PEFT**: 在原子召回和多步函数调用任务上，TokMem持续优于RAG，并以显著更少的参数匹配或超越parameter-efficient fine-tuning (PEFT)。

## Method

TokMem的核心思想是将每个可重用的任务程序编码为一个独特的、可训练的记忆token。

1.  **程序编译**: 每个特定的任务（例如，“总结文档”或“调用API”）被编译成一个唯一的记忆token。这个token被设计成能够捕捉该任务的语义和执行逻辑。
2.  **记忆token作为控制信号**: 在LLM生成过程中，这些记忆token被注入到模型的输入序列中。它们充当生成控制信号，直接引导LLM生成与该任务程序相关的输出。
3.  **冻结LLM骨干**: 训练过程中，LLM的骨干参数保持冻结。只有这些记忆token的嵌入（embeddings）是可训练的。
4.  **持续添加新程序**: 由于新程序的知识仅存储在新的记忆token中，因此可以持续添加新任务，而不会影响或遗忘已学习的现有任务。
5.  **恒定大小开销**: 无论有多少个记忆token，每次推理时只需处理相关的记忆token，因此对上下文长度的开销是恒定的。

## Comparison with Previous Work

- **vs. Retrieval-Augmented Generation (RAG)**:
    - RAG通过检索相关文档来扩充prompt，但每次查询都需要重新处理整个上下文，导致重复的计算开销。
    - TokMem将程序知识编码为固定大小的token，避免了重复的上下文处理，且在性能上持续优于RAG。
- **vs. Parameter-Efficient Fine-Tuning (PEFT)**:
    - PEFT通过微调LLM的少量参数来适应新任务，但可能面临灾难性遗忘问题，且难以模块化地添加新任务。
    - TokMem通过训练独立的记忆token来存储程序知识，避免了对LLM骨干的修改，实现了更好的模块化和持续学习能力，并以更少的参数达到或超越PEFT的性能。
- **vs. Standard Prompting**:
    - 标准prompt需要为每个任务手动设计和重复输入长文本，效率低下且难以复用。
    - TokMem将复杂程序抽象为单个token，大大简化了控制，并降低了推理时的计算成本。

## Motivation

Large language models (LLMs) are typically controlled via prompts, which suffer from inefficiencies as they must be repeatedly re-processed for every new query.
Furthermore, modular reuse of prompts is challenging, making it difficult to encapsulate and independently utilize specific task logic.
This limitation restricts LLMs' application in scenarios requiring numerous reusable and composable tasks.

## Contribution

- **Procedural Memory Framework TokMem**: Introduction of a framework that compiles each reusable task procedure into a single trainable memory token.
- **Constant-Size Overhead Control**: Each memory token serves as both a procedure index and a generation control signal, steering generation with constant-size overhead to enable targeted behaviors.
- **Frozen LLM Backbone**: TokMem keeps the backbone LLM frozen when adding new procedures, storing procedural knowledge entirely in these dedicated units, thus avoiding interference with existing knowledge.
- **Outperforms RAG and PEFT**: TokMem consistently outperforms Retrieval-Augmented Generation (RAG) and matches or exceeds parameter-efficient fine-tuning (PEFT) with substantially fewer trainable parameters on both atomic recall and multi-step function-calling tasks.

## Method

The core idea of TokMem is to encode each reusable task procedure into a unique, trainable memory token.

1.  **Procedure Compilation**: Each specific task (e.g., "summarize document" or "call API") is compiled into a unique memory token. This token is designed to capture the semantics and execution logic of that task.
2.  **Memory Token as Control Signal**: During LLM generation, these memory tokens are injected into the model's input sequence. They act as generation control signals, directly guiding the LLM to produce outputs relevant to the task procedure.
3.  **Frozen LLM Backbone**: During training, the backbone parameters of the LLM remain frozen. Only the embeddings of these memory tokens are trainable.
4.  **Continuous Addition of New Procedures**: Since knowledge for new procedures is stored only in new memory tokens, new tasks can be continually added without affecting or causing catastrophic forgetting of existing learned tasks.
5.  **Constant-Size Overhead**: Regardless of the number of memory tokens, only the relevant memory tokens need to be processed during inference, resulting in a constant overhead to the context length.

## Comparison with Previous Work

- **vs. Retrieval-Augmented Generation (RAG)**:
    - RAG augments prompts by retrieving relevant documents, but each query requires re-processing the entire context, leading to repeated computational overhead.
    - TokMem encodes procedural knowledge into fixed-size tokens, avoiding repeated context processing, and consistently outperforms RAG in performance.
- **vs. Parameter-Efficient Fine-Tuning (PEFT)**:
    - PEFT adapts LLMs to new tasks by fine-tuning a small number of parameters, but can suffer from catastrophic forgetting and lacks modularity for adding new tasks.
    - TokMem stores procedural knowledge by training independent memory tokens, avoiding modifications to the LLM backbone, enabling better modularity and continual learning, and achieving comparable or superior performance to PEFT with fewer parameters.
- **vs. Standard Prompting**:
    - Standard prompting requires manually designing and repeatedly inputting lengthy text for each task, which is inefficient and difficult to reuse.
    - TokMem abstracts complex procedures into single tokens, significantly simplifying control and reducing computational costs during inference.
